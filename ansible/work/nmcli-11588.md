# Bug Analysis — Issue #11588

**Title:** `nmcli` wrong/missing attributes when in check/diff  
**Upstream:** https://github.com/ansible-collections/community.general/issues/11588  
**Module file:** `plugins/modules/nmcli.py`  
**Working branch:** `fix/nmcli-bond-arp-diff-11588`  
**PR:** https://github.com/ansible-collections/community.general/pull/12085  
**Status:** PR open (May 2026)

---

## The Problem

When managing a **bond** connection with `arp_interval` and/or `arp_ip_target`, the module reports **changed on every run** even when those values are already configured in NetworkManager.

This breaks:

- **Check mode** (`--check`) — false positives block safe dry-runs
- **Diff mode** (`--diff`) — noisy, misleading diffs in CI
- **Idempotency** — automation looks like it is constantly drifting

### Example task

```yaml
- community.general.nmcli:
    type: bond
    conn_name: bond0
    ifname: bond0
    mode: active-backup
    arp_interval: 100
    arp_ip_target: 192.168.1.1
    state: present
```

After the first run configures the bond, a second run should return `changed: false`. Before the fix, it returned `changed: true` for the ARP options.

---

## Root Cause

The module has two code paths that must agree on **property key names**:

| Code path | What it does | Key format used (before fix) |
|-----------|--------------|------------------------------|
| `show_connection()` | Parses `nmcli con show` output | Reads `bond.options` → keys like `arp_interval`, `arp_ip_target` (underscores) |
| `connection_options()` | Builds desired state for compare/modify | Used `arp-interval`, `arp-ip-target` (hyphens) |

`_compare_conn_params()` compares desired keys against `conn_info` keys. Because `arp-interval` ≠ `arp_interval`, the module never found a match and always treated ARP settings as new/changed.

### Related issue: bond MTU false positive

Bond connections with `mtu` unset also triggered a false change: `802-3-ethernet.mtu` compared `None` (not in show output) against `"auto"` (default when detecting changes). The module already handled this for `dummy` connections but not for `bond` / `bond-slave`.

---

## Fix Strategy

### 1. Align bond ARP key names

In `connection_options()` for `type == "bond"`:

```python
# Before (buggy)
"arp-interval": self.arp_interval,
"arp-ip-target": self.arp_ip_target,

# After (fixed)
"arp_interval": self.arp_interval,
"arp_ip_target": self.arp_ip_target,
```

These names match how `show_connection()` parses `bond.options`.

### 2. Set ARP options via `+bond.options` on create/modify

In `connection_update()`, route ARP options through the same mechanism as `xmit_hash_policy` and `fail_over_mac`:

```python
if key in ("xmit_hash_policy", "fail_over_mac", "arp_interval", "arp_ip_target"):
    cmd.extend(["+bond.options", f"{key}={value}"])
```

NetworkManager stores these as bond options, not as top-level hyphenated properties.

### 3. Extend MTU idempotency to bond types

In `_compare_conn_params()`, treat missing MTU the same as `dummy` when `mtu` param is unset:

```python
self.type in ("bond", "bond-slave", "dummy")
```

## Dependent Components (verified)

| Component | Role | Impact |
|-----------|------|--------|
| `connection_options()` | Builds desired-state dict | **Changed** — ARP keys fixed |
| `connection_update()` | Runs `nmcli con add/modify` | **Changed** — ARP via `+bond.options` |
| `show_connection()` | Parses existing connection | **Unchanged** — already correct |
| `is_connection_changed()` | Orchestrates compare | **Unchanged** — uses fixed options |
| `_compare_conn_params()` | Key-by-key diff | **Changed** — bond MTU false positive |
| `settings_type()` | Type hints for parsing | **Unchanged** — ARP not special-cased |
| Module DOCUMENTATION | User-facing params | **Unchanged** — `arp_interval` / `arp_ip_target` already documented |
| Integration tests | — | **None exist** for nmcli (unit tests only) |

No other modules or plugins import nmcli internals. The fix is self-contained.
