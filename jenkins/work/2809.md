## Problem

[PR #1553](https://github.com/jenkinsci/kubernetes-plugin/pull/1553) passes `--noReconnectAfter` to the inbound agent so the **jnlp** JVM exits after it cannot reconnect (e.g. build finished). That works when the pod has **only jnlp**.

Production-style agents often add sidecars:

- `docker` / `docker-daemon` (DinD)
- `sleep infinity` helpers
- custom tools containers

**Actual behaviour:** jnlp exits ŌåÆ sidecar still **Ready** ŌåÆ pod stays **Running** ŌåÆ cluster resource leak until manual cleanup.

**Related work logs:** multi-container pods (`jnlp`, `docker`, `docker-daemon`) on corp Jenkins ŌĆö same shape as this bug class (agent timeout / pending containers), though #2809 is specifically post-build cleanup not ImagePullBackOff.

---

## Root cause (working theory)

Kubernetes pod phase stays **Running** while any container is ready. `-noReconnectAfter` only stops **jnlp**; the plugin does not delete the pod or stop sibling containers when the agent disconnects.

Issue discussion points at [pod lifecycle / restart policy](https://kubernetes.io/docs/concepts/workloads/pods/pod-lifecycle/#restart-policy) limits and missing liveness coordination across containers.

**Code areas to study:**

- `KubernetesLauncher` ŌĆö agent launch and reconnect flags
- `KubernetesSlave` / `_terminate()` ŌĆö pod deletion paths
- How #1553 injects `--noReconnectAfter` (single-container assumption)

---

## Expected fix

When the **Jenkins agent container** terminates permanently:

1. Delete the **pod**, or  
2. Stop **all** containers in the pod template, or  
3. Add **liveness** semantics so a dead jnlp fails the pod

Must not break single-container pods (regression tests required).
